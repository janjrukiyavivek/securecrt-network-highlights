Network Engineer Ultimate

A SecureCRT keyword-highlighting pack generated and validated from data — not hand-edited — covering the multi-vendor CLI output a network engineer actually sees every day.

SecureCRT — R1 
R1# show ip bgp summary
BGP router identifier 10.0.0.1, local AS number 65001
Neighbor        V    AS  MsgRcvd  MsgSent   TblVer  InQ OutQ  Up/Down  State/PfxRcd
10.0.0.2        4 65002     1204     1198        0    0    0 3d04h          Active
10.0.0.3        4 65003        0        0        0    0    0 never          Idle

%LINK-3-UPDOWN: Interface GigabitEthernet0/1, changed state to down
%OSPF-5-ADJCHG: Process 1, Nbr 10.0.0.3 on Gi0/2 from FULL to DOWN

Severity-tiered by design: %LINK-3 and a bare "down" state read as critical (red), %OSPF-5 reads as a warning (amber) — matching SecureCRT's own top-to-bottom, first-match-wins highlighting engine.

Why it's different

Most highlight files accumulate duplicate and conflicting rules over years of manual edits — the same word ends up colored two different ways depending on which rule was pasted in last, and nobody notices because SecureCRT silently uses whichever rule comes first.

This pack is generated from tools/build.py, which refuses to build if a regex is duplicated or fails to compile, and tools/validate_highlights.py independently checks the output file's structure. Every cross-section word collision found during development — e.g. "Active" meaning three different things in BGP vs. HSRP vs. generic interface state — was deliberately resolved and documented inline in the build script, not left as a silent bug.

Not covered

  • Palo Alto / Fortinet / FTD CLI specifics — only generic, cross-vendor tokens (ACL, IPsec, IKE) highlight; vendor-specific command syntax isn't covered yet.
  • Cisco ACI — not covered.
  • Context beyond the current line — SecureCRT highlighting is per-line regex matching; it cannot tell BGP's "Active" state apart from an HSRP "Active" router by looking at surrounding lines.

Install

  1. Run python3 tools/build.py to (re)generate build/Network Engineer Ultimate.ini.
  2. Run python3 tools/validate_highlights.py to confirm it's well-formed.
  3. In SecureCRT: Options → Session Options → Terminal → Emulation → Keyword Highlighting → Import, and select the generated .ini file.

Or skip the build step entirely — use the Download .ini link at the top of this page to grab the latest generated file directly.

Design principles

  • Designed for production troubleshooting, not a colorful terminal. A rule only exists if it highlights a condition that needs attention or speeds up reading dense CLI output.
  • Highlight the exception, not the norm: counters only light up when non-zero; "down" only matches when it's actually describing link/protocol state.
  • Specific regex before generic regex, always — SecureCRT applies the first matching rule top-to-bottom per character, so ordering is the mechanism that decides which color wins a contested word.
  • Generated from data (tools/build.py), not hand-edited, so duplicate and conflicting rules are caught by the build itself instead of accumulating silently over years.
  • Scope is intentionally limited to platforms validated against real sample output — new vendor support is added based on real-world usage, not completeness for its own sake.

Maintaining

All rules live as data in tools/build.py (see docs/REGEX_GUIDE.md for conventions). To add or change a rule:

  1. Edit tools/build.py — it fails loudly on a duplicate or invalid regex rather than silently producing a broken file.
  2. Run python3 tools/validate_highlights.py to check the generated file's structure.
  3. Run python3 tools/test_samples.py to see how the new rule renders against realistic multi-vendor show-command output, and confirm nothing else's coloring shifted as a side effect.

Changelog

v1.2

  • Added tools/test_samples.py, simulating SecureCRT's actual top-to-bottom, first-match-wins behaviour against realistic multi-vendor output — surfaced and fixed four real bugs (BGP best-path marker misread as a login prompt, "line protocol is down" losing its colour to a generic rule, a Route Target regex colliding with BGP uptime timestamps, and EVPN bracket-notation route types not matching).
  • Added a curated Sophos Firewall (SFOS) section: platform terms, daemon/service names, firewall actions, VPN/HA/routing events, security modules, and audit terms.
  • Added a tcpdump-output section: TCP flag brackets coloured by severity, seq/ack/win/length fields, ICMPv6, and microsecond timestamps.
  • Removed the legacy source/ directory of reference files — no longer presented as a merge of prior highlight packs.

v1.1

  • Added Cisco bug ID (CSCxxNNNNN) and CVE highlighting, Nexus vPC, StackWise/VSS.
  • Added EVPN route types, scoped to avoid colliding with Nexus vPC's unrelated wording.
  • Split syslog severity into four colour tiers instead of one flat colour for every facility line.
  • Made interface "down" state context-aware instead of matching the bare word anywhere on the line.
